# Poland Urban Transit & Real Estate Analysis Platform (2026)

## 1. Project Mission and Analytical Scope
The primary mission of this platform is to provide an empirical, high-fidelity quantification of the causal relationship between public transport accessibility and residential property values across 29 major Polish agglomerations. By integrating high-resolution transit data (GTFS), comprehensive infrastructure context (OSM), transactional real estate registries (RCN), and demographic grids (GUS NSP 2021), the system enables advanced modeling of Transit-Oriented Development (TOD) premiums.

This platform is not merely a data aggregator; it is a specialized spatial engineering engine designed to eliminate "spatial noise" and provide a "clean" environment for econometric modeling.

## 2. Global Data Inventory: Pillar of Truth

### 2.1 Administrative and Reference Data (data/poland/)
- **admin/powiaty.json**: The master registry of Polish administrative boundaries at the county (powiat) level. Sourced from the National Register of Boundaries (PRG). This is the "spatial anchor" used for all TERYT-to-City mappings and WFS query targeting.
- **admin/all_stops_smart.gpkg**: A consolidated spatial database containing 55,933 validated transit nodes (100% Hub Coverage). These stops have been meticulously filtered via the "Urban Gravity Engine" to ensure they represent urban and suburban movement, excluding long-distance national rail artifacts.
- **osm/poland-latest.osm.pbf**: A 1.9GB binary source file containing the complete OpenStreetMap dataset for Poland. This file acts as the single source of truth for all local infrastructure extractions, ensuring temporal and structural consistency across all 29 cities.
- **population/nsp2021_grid250m.gpkg**: A high-resolution (250m) demographic grid from the 2021 National Census. It provides the "denominator" for accessibility metrics, allowing for population-weighted transit coverage analysis.

### 2.2 City-Centric Analytical Hubs (data/cities/)
The system utilizes a modular "Hub" architecture. Each of the 29 directories represents a self-contained analytical ecosystem, allowing for independent processing, validation, and recovery:

- **gtfs/**: Contains unpacked and synchronized GTFS feeds from multiple local operators (e.g., ZTM Warszawa, MPK Kraków, SKM Trójmiasto).
- **rcn/**: The core registry of real estate transactions. Contains individual GML files per county and the consolidated `transactions.gpkg` database.
- **osm_full.gpkg**: A relational infrastructure database containing all buildings, roads, sidewalks, and POIs, with specialized attribute preservation for urban density modeling.
- **smart_stops.gpkg**: The city-specific subset of transit nodes, verified for metropolitan relevance using the gravity algorithm.
- **transport_zone.gpkg**: The definitive analysis mask generated by merging 1,500m pedestrian buffers around validated stops. All secondary data (OSM, RCN) is strictly clipped to this geometry.
- **rcn_targets.json**: A machine-generated manifest identifying every administrative TERYT code that intersects with the city's transport zone.

## 3. Core Methodology: The Urban Gravity Engine (KPP 2026)
To solve the "Train Plague" problem (where nationwide rail networks inflate urban boundaries), the project implements a proprietary three-stage spatial filtering algorithm:

1. **Urban Core Identification**: We first aggregate all "Class A" transit nodes (Bus, Tram, Metro, Trolleybus). These form the high-density urban skeleton.
2. **Gravity-Based Rail Clipping**: "Class B" nodes (Regional rail stations) are retained if and only if they are located within a 5,000-meter radius of any Class A node. This ensures we capture suburban commuters while dropping rural stations that do not contribute to metropolitan TOD.
3. **Analytical Perimeter Generation**: A 1,500-meter "pedestrian reach" buffer is calculated around all validated "Smart Stops". This geometry is topologically verified and simplified (5m tolerance) to serve as the master spatial mask for all subsequent extractions.

## 4. Engineering Pipelines (scripts/)

### 4.1 Production Pipelines (scripts/core/)
- **fetch_gtfs_enterprise.py**: A multi-threaded (ThreadPoolExecutor) synchronizer. It manages over 85 diverse GTFS sources, categorizing them into city hubs and enforcing a strict blacklist for national rail operators (PKP IC, PolRegio).
- **collect_stops_smart.py**: The implementation of the Urban Gravity algorithm. It processes raw GTFS files, performs spatial clustering, and generates the `smart_stops.gpkg` and `transport_zone.gpkg`.
- **prepare_osmium_config.py**: A diagnostic tool that calculates Bounding Boxes (BBOX) for each city Hub and generates optimized JSON configurations for the Osmium binary engine.
- **extract_osm_bulletproof.py**: A high-performance, sequential ETL pipeline. It uses C++ Osmium for binary clipping and GDAL/OGR for relational conversion. It is designed for ultra-low RAM environments (RSS < 2GB) to prevent system instability during large-scale processing.
- **identify_rcn_targets.py**: Performs local spatial joins to map city perimeters to administrative TERYT codes, eliminating redundant or empty WFS queries.
- **fetch_rcn_ogr.py**: A robust harvester utilizing the GDAL/OGR WFS driver. It handles paging (2000 records/page), XML integrity, and server-side filtering (`-where`) to ensure 100% data retrieval.
- **consolidate_rcn.py**: Merges fragmented GML transaction data into a single, optimized GeoPackage with R-Tree spatial indexing.
- **osmconf.ini**: A specialized GDAL configuration that forces "Take Everything" attribute extraction and promotes closed ways to Multipolygons for precise floor area calculations.

### 4.2 Utility & Audit Suite (scripts/utils/)
- **spatial_integrity_audit.py (SICA)**: A structural reporter that analyzes every Hub, verifying metadata integrity and spatial coverage (OSM vs RCN vs Population).
- **final_data_validation.py**: A geometrical validator that enforces city area limits (< 6,000 km²) and coordinate system consistency (EPSG:4326 for storage, EPSG:2180 for metrics).
- **deep_data_audit_v3.py**: Counts millions of points, lines, and polygons across the 29 hubs to ensure zero data loss during conversion.

### 4.3 Testing & Diagnostics (scripts/testing/)
- **benchmark_osm.py**: Tracks Peak RSS and CPU usage during heavy spatial tasks to prevent OOM failures.
- **test_rcn_single.py**: A Proof-of-Concept script for WFS endpoint validation and coordinate order verification.

## 5. Data Dictionary: The `all_tags` Logic
To ensure the platform is future-proof, all OpenStreetMap attributes are preserved. While common columns (like `building`, `highway`, `amenity`) are indexed, the `all_tags` column stores the complete metadata in a JSON-like format:
- **Usage:** `SELECT * FROM multipolygons WHERE all_tags LIKE '%"building:levels"=>"10"%'`
- **Benefit:** Allows for arbitrary future queries (e.g., building materials, roof types, opening hours) without re-extracting data from source PBF files.

## 7. Execution Workflow (Standard Operating Procedure)
To initialize the system from scratch, the following sequence must be maintained:
1. **Sync Transit:** `python3 scripts/core/fetch_gtfs_enterprise.py`
2. **Define Boundaries:** `python3 scripts/core/collect_stops_smart.py`
3. **OSM Configuration:** `python3 scripts/core/prepare_osmium_config.py`
4. **ETL Infrastructure:** `python3 scripts/core/extract_osm_bulletproof.py`
5. **Map TERYT Targets:** `python3 scripts/core/identify_rcn_targets.py`
6. **Harvest Transactions:** `python3 scripts/core/fetch_rcn_ogr.py`
7. **Consolidate & Verify:** `python3 scripts/core/consolidate_rcn.py` && `python3 scripts/utils/spatial_integrity_audit.py`

## 8. Technical Stack and Performance
- **Language:** Python 3.12+ (GeoPandas, Pandas, SQLAlchemy)
- **Spatial Engines:** Osmium Tool (C++), GDAL/OGR 3.8+ (C++)
- **Storage:** GeoPackage (GPKG) with R-Tree indexing, SQLite 3.x
- **Performance Strategy:** Sequential BBOX extraction + Parallel WFS Harvesting.
